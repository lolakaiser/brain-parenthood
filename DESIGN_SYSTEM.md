# Brain Parenthood Design System
**Version 2.0 - Workplace Resilience & Team Learning Platform**

---

## Design Philosophy

Brain Parenthood is designed to create a psychologically safe learning environment for workplace resilience training. Every design decision supports four core user feelings:

1. **"I know exactly where I am in the program"** → Clear hierarchy, consistent navigation, visible progress
2. **"This is safe to engage with"** → Calm colors, gentle animations, non-judgmental language
3. **"I can do this even on a low-energy day"** → Low cognitive load, clear CTAs, minimal choices
4. **"My team is moving forward together"** → Team indicators, collective progress, shared milestones

---

## Visual Identity

### Brand Personality
- **Grounded**: Like a steady guide, not a drill sergeant
- **Warm**: Professional but approachable
- **Clear**: Structured without being rigid
- **Encouraging**: Progress-focused, not perfection-focused

### Design Principles
1. **Clarity over cleverness** - Always choose the obvious over the clever
2. **Consistency builds trust** - Predictable patterns reduce cognitive load
3. **Progress over perfection** - Show movement, not just achievement
4. **Calm by default** - No jarring transitions, loud colors, or aggressive prompts

## Typography

### Font Families

**Headlines & UI: Space Grotesk**
- Modern, geometric, friendly
- Excellent readability at all sizes
- Feels structured but not rigid
- Weights: 400 (Regular), 500 (Medium), 600 (SemiBold), 700 (Bold)

**Body Text: Inter**
- Designed specifically for screens
- High x-height for easy reading
- Neutral and professional
- Weights: 400 (Regular), 500 (Medium), 600 (SemiBold)

**Reasoning**: Space Grotesk for headlines provides personality and structure, while Inter for body text ensures maximum readability for long-form content. This combination is modern, professional, and optimized for digital reading.

| `space-1` | 4px | Tight spacing, icon padding |
| `space-2` | 8px | Small gaps, button padding |

| `space-4` | 16px | Standard padding in cards, sections |
| `space-5` | 20px | Comfortable gaps between elements |
| `space-6` | 24px | Section padding |
| `space-8` | 32px | Major section gaps |
| `space-10` | 40px | Large spacing between content blocks |
| `space-12` | 48px | Section padding on desktop |
| `space-16` | 64px | Hero section padding |
| `space-20` | 80px | Major page section spacing |
| `space-24` | 96px | Large page section spacing |

**Reasoning**: Consistent spacing creates visual rhythm and reduces decision fatigue. The 4px base unit ensures all elements align to a grid.

| `radius-sm` | 8px | Small elements (badges, tags) |

## UI Patterns for Psychological Safety

### 1. Clear Orientation Pattern
**Problem**: Users feel lost or uncertain about their location
**Solution**:
- Always show: Current module, week number, step in process
- Breadcrumb navigation on every screen
- Progress indicators visible at all times
- "You are here" markers in navigation

**Example**:
```
[Home > Module 3 > Week 3 > Baseline Assessment]
Progress: 3 of 12 modules complete
```

---

### 2. Gentle Feedback Pattern
**Problem**: Errors feel like failures in a mental health context
**Solution**:
- Never use "Error" or "Failed" language
- Instead: "Let's try that again" or "Almost there"
- Show what's needed, not what's wrong
- Use supportive language: "This field needs a bit more info"

**Example**:
```
❌ BAD: "Error: Password must be 8 characters"
✅ GOOD: "Your password needs at least 8 characters for security"
```

---

### 3. Low-Energy Mode Pattern
**Problem**: Users on low-energy days can't engage with complex UIs
**Solution**:
- Always show a "Continue where you left off" option
- Auto-save all progress (no manual saves required)
- Break tasks into < 5-minute chunks
- Show estimated time for each section
- Allow "Save and finish later" at any point

**Example**:
```
[Continue: Module 3, Step 2]
"Welcome back! You were working on your team goals. This should take about 3 minutes."
[Resume] [Start Over]
```

---

### 4. Team Progress Pattern
**Problem**: Individual users feel isolated
**Solution**:
- Show team member count on every screen
- Display aggregate (not individual) team progress
- Use "We" language: "We've completed 3 modules"
- Show team milestones: "Your team unlocked Week 4!"
- Never compare individuals

**Example**:
```
👥 Your Team (5 members)
📊 Team Progress: 3.2 modules average
🎉 Next milestone: Week 4 (2 more modules)
```

---

### 5. Non-Judgmental Metrics Pattern
**Problem**: Metrics can feel like performance reviews
**Solution**:
- Focus on progress, not achievement
- Use neutral language: "Current stress level" not "How stressed are you?"
- Show change over time, not absolute numbers
- Frame everything as learning: "This helps us understand your baseline"
- No "good" or "bad" labels on data

**Example**:
```
❌ BAD: "Your stress is HIGH (8/10)"
✅ GOOD: "Your baseline: 8/10. We'll track how this changes."
```

---

### 6. Cognitive Load Reduction Pattern
**Problem**: Too many choices create anxiety
**Solution**:
- Maximum 2-3 choices per screen
- One primary CTA per section
- Progressive disclosure: Show advanced options only when needed
- Default selections for most choices
- "Recommended" labels for suggested options

**Example**:
```
[Continue to Next Module] ← Primary
[Review Previous Module] ← Secondary
... (hidden menu)
  - Download Progress
  - Settings
  - Help
```

---

### 7. Celebration Without Pressure Pattern
**Problem**: Gamification can feel childish or create pressure
**Solution**:
- Quiet celebrations: Subtle animations, soft colors
- Acknowledge progress without comparing
- Optional milestones (not required)
- Use mature language: "Module complete" not "You did it!"
- No public leaderboards or rankings

**Example**:
```
✓ Module 3 Complete
"You've built a strong foundation in team communication.
Ready to continue when you are."
[Continue to Module 4] [Return to Dashboard]
```

---

### 8. Consistency Breeds Safety Pattern
**Problem**: Unfamiliar interfaces create anxiety
**Solution**:
- Every screen follows the same layout:
  1. Navigation at top
  2. Progress indicator
  3. Content area with clear heading
  4. Primary CTA bottom-right
  5. Secondary actions bottom-left
- Same colors mean the same things everywhere
- Icons always paired with text labels
- Animations are predictable and gentle

---

### 9. Accessible by Default Pattern
**Problem**: Accessibility is often an afterthought
**Solution**:
- Minimum 44px touch targets
- 4.5:1 color contrast (WCAG AA)
- All actions keyboard-accessible
- Screen reader labels on all interactive elements
- Never rely on color alone (use icons + text)
- Reduced motion option available

---

### 10. Privacy & Safety Pattern
**Problem**: Sharing mental health data feels vulnerable
**Solution**:
- Clear data ownership statements
- "Only you can see this" labels on personal data
- "Your team sees aggregate data only" on team views
- Easy opt-out of any optional sharing
- Visible privacy controls

**Example**:
```
🔒 Your Baseline Data (Private)
This data is used to show your personal progress.
Your team admin cannot see individual responses.
[Learn about privacy]
```

## Animation Principles

### Speed & Easing
- **Fast actions** (100-150ms): Hovers, focus states
- **Standard actions** (200-300ms): Cards, buttons, modals opening
- **Slow actions** (400-500ms): Page transitions, celebration animations
- **Easing**: Use `ease-out` for entrances, `ease-in` for exits, `ease-in-out` for continuous movement

### Animation Types
1. **Fade**: Opacity 0 → 1 (or reverse)
2. **Slide**: translateY(-10px) → 0
3. **Scale**: Scale 0.95 → 1
4. **Combined**: Fade + slide or fade + scale for entrances

### When to Animate
- ✅ User-initiated actions (clicks, hovers)
- ✅ Content appearing/disappearing
- ✅ Progress changes
- ✅ Feedback to user actions
- ❌ Auto-playing content
- ❌ Looping animations (except loading)
- ❌ Animations that can't be disabled

### Reduced Motion
Always respect `prefers-reduced-motion: reduce`:
```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

**Reasoning**: Animations should guide attention and provide feedback, not distract or overwhelm. Fast, purposeful animations feel responsive. Respecting user preferences is critical for accessibility.

## Responsive Design

### Breakpoints
- **Mobile**: 0-639px (single column, stacked layout)
- **Tablet**: 640-1023px (flexible layout, some side-by-side)
- **Desktop**: 1024-1535px (full features, multi-column)
- **Large Desktop**: 1536px+ (max-width container, generous spacing)

### Mobile-First Approach
1. Design for mobile first (smallest screen)
2. Add complexity as screen size increases
3. Never hide critical features on mobile
4. Touch targets minimum 44x44px
5. Font size minimum 16px (prevents iOS zoom)

### Layout Strategy
- **Mobile**: Single column, full-width cards
- **Tablet**: 2-column grid where appropriate, flexible cards
- **Desktop**: 3-column grid (dashboard), sidebar + content layouts
- **Max width**: 1400px container to prevent overly wide content

**Reasoning**: Mobile-first ensures core experience is accessible. Progressive enhancement adds richness without removing functionality.

## Accessibility Standards

### WCAG 2.1 AA Compliance
1. **Color contrast**: 4.5:1 for body text, 3:1 for large text/icons
2. **Keyboard navigation**: All interactive elements must be keyboard accessible
3. **Focus indicators**: Clear 2px outline on focus (Primary 500)
4. **Alt text**: All images have descriptive alt text
5. **ARIA labels**: All buttons/links have clear labels
6. **Semantic HTML**: Use proper heading hierarchy (h1 → h2 → h3)

### Screen Reader Support
- Landmark regions: header, nav, main, footer, aside
- Skip to content link (hidden but keyboard-accessible)
- Status announcements for dynamic content (aria-live)
- Clear form labels and error messages

### Keyboard Navigation
- Tab order follows visual order
- Enter/Space activate buttons and links
- Escape closes modals and dropdowns
- Arrow keys navigate lists and menus

**Reasoning**: Accessibility is not optional in a mental health product. Everyone deserves access to resilience training, regardless of ability.
